GemMarket — Full Component Workflow Documentation
Tech Stack: React Native (Frontend) · Node.js + Express.js (Backend) · MongoDB (Database) · JWT (Authentication) · Bcrypt (Password Hashing) · Multer (File Uploads) · Stripe (Payment Gateway) · AsyncStorage (Token Storage) · Hosted Backend (Render / Railway / AWS)
Group Component — User Authentication
The authentication module is the entry point of the entire application. Every person who uses the app — whether a customer or an admin — must go through this module first.
When a new user opens the app for the first time, they are taken to the registration screen built in React Native. They enter their full name, email address, and a password. The form is validated on the frontend before submission. The app then sends this data to the backend at the POST /api/auth/register endpoint, built with Node.js and Express.js. The backend validates that the email is not already taken and that all required fields are filled. It then runs the password through bcrypt, which converts it into a hashed string before saving it to the MongoDB database. The actual password is never stored anywhere — only the hash. Once saved, the user account is created with a role of "customer" by default.
When a returning user logs in, they enter their email and password on the login screen. The app sends this to POST /api/auth/login. The backend finds the user's record in MongoDB by email, then uses bcrypt to compare the submitted password against the stored hash. If they match, the server generates a JWT token — a digitally signed string that proves the user is who they say they are — and sends it back to the app. The app stores this token on the device using AsyncStorage so the user stays logged in between sessions without needing to re-enter their credentials.
Every time the app makes a request to a protected endpoint, it attaches the JWT token in the request header. The backend has a middleware function built in Express.js that runs before any protected route. This middleware reads the token, verifies its signature and checks it has not expired, and then either allows the request to continue or returns a 401 Unauthorized error if the token is invalid or missing.
Admin accounts are created directly in the MongoDB database during setup. The admin role is stored in the user's record and is encoded inside the JWT token. When the admin logs in, the app detects the admin role from the token and shows the admin panel instead of the regular customer interface. Any endpoint that requires admin access checks the role from the token and blocks the request if the user is a regular customer.

1. Inventory Management
The inventory module is a backend stock register that the admin uses to keep track of all gems the business owns. It is built using Node.js, Express.js, and MongoDB, and is entirely internal — customers never interact with the inventory directly.
When the admin wants to add a new gem to the business stock, they open the inventory screen in the React Native admin panel and fill in the gem's basic properties: name, type or category, colour, carat amount, and stock quantity. This is submitted to POST /api/inventory and saved as a new document in the MongoDB database. There are no images, prices, or descriptions at this stage — inventory is purely a factual record of what exists in stock.
When the admin needs to update a gem — for example, if the carat measurement was entered incorrectly or the stock count changes — they select the gem from the inventory list and edit the relevant fields. This sends a PUT /api/inventory/:id request to update that specific document in MongoDB.
When the admin wants to remove a gem from stock — either because it was physically disposed of or it was entered by mistake — they can delete it manually, which calls DELETE /api/inventory/:id.
The most important automated action in this module is the stock update that happens when a gem is sold. When a customer completes a purchase through the marketplace, wins a bid, or has a negotiated offer accepted, the system automatically decrements the stock count for that gem in the inventory. If the stock count reaches zero, the gem is marked as unavailable. This keeps the inventory always accurate without the admin having to manually update it after every sale.
2. Learning Hub
The Learning Hub is an educational content module where the admin publishes articles and guides about gemstones for customers to read. It is built using Node.js, Express.js, and MongoDB on the backend, with Multer handling cover image uploads, and React Native screens on the frontend.
When the admin wants to add a new article, they open the Learning Hub management screen in the admin panel. They fill in a title, select a category from a predefined list — such as Gem Types, Buying Guide, Grading & Quality, or Care & Maintenance — write the body content, and upload a cover image. The image is handled using Multer on the backend, which validates the file type and size before storing it and returning the image URL to be saved with the article record in MongoDB. Submitting the form sends a POST /api/learning request to the backend, which saves the article and makes it immediately visible to customers.
The admin can view all published articles from a management screen in the admin panel, which fetches them from GET /api/learning. From here the admin can select any article to edit — for example to correct information or swap the cover image — which sends a PUT /api/learning/:id request. If an article is no longer relevant, the admin can delete it via DELETE /api/learning/:id.
On the customer side, the Learning Hub is accessible from the main navigation of the React Native app. The hub home screen fetches all published articles from GET /api/learning and displays them as a scrollable list, each card showing the cover image, title, and category tag. Customers can filter articles by category using query parameters sent to the same endpoint.
When a customer taps on an article, the full article page is fetched from GET /api/learning/:id and displays the cover image, title, category, and full body content. At the bottom of the article page, if the article category matches a gem type listed in the marketplace, the app displays a View Related Gems button that takes the customer directly to the marketplace filtered by that gem category. This connection is handled on the frontend by passing the article's category as a filter parameter to the marketplace endpoint — no additional backend relationship is required.
The article document in MongoDB stores the title, category, body content, image URL, and the date it was published. The category field is validated on both the frontend form and the backend controller to ensure only accepted values are saved. All write routes are protected by JWT middleware so that only admin users can create, update, or delete articles, while customers and unauthenticated users can only read them.
3. Direct Marketplace & Negotiation System
The marketplace is the customer-facing shop and the core commercial module of the application. It is built with Node.js, Express.js, and MongoDB on the backend, with Multer used for gem photo and video uploads, and React Native screens handling the customer and admin interfaces.
When the admin wants to put a gem up for sale, they open the create listing screen in the admin panel. They select a gem from the inventory — this pulls in the basic properties like name, colour, and carats automatically. The admin then adds the selling price, a written description, and uploads photos and optionally a video of the gem using Multer. At this point, the admin also decides whether this gem should accept customer offers or be sold at a fixed price only. If they toggle it to open for negotiation, the listing will show a Make an Offer button to customers. Once everything is filled in, the admin submits and the listing is published to POST /api/marketplace and becomes visible to all customers.
When a customer opens the app, they land on the marketplace home screen which displays all available gem listings fetched from GET /api/marketplace. Each listing shows the gem photo, name, basic properties, and price. Customers can search by name or filter by category, type, or price range using query parameters.
When a customer taps on a listing, they go to the gem detail screen fetched from GET /api/marketplace/:id. This page shows all photos, the video if one was added, the full description, all gem properties, and the price. From here the customer can either buy the gem immediately at the listed price, or if the listing is open for negotiation, they will also see a Make an Offer button.
When a customer chooses to buy immediately, they confirm the purchase and are directed to the payment module to complete the transaction. Once payment is confirmed, an order is automatically created in the order tracking module and the gem is simultaneously removed from the inventory. The listing is then closed and no longer visible to other customers.
When a customer taps Make an Offer, they are taken to the offer submission screen where they type in the price they are willing to pay and submit it. This sends a POST /api/offers request to the backend, saving the offer linked to the gem and the customer with a status of Pending. The customer can view their submitted offers and statuses from their account page.
On the admin side, all incoming offers are listed in an offers management screen, grouped by gem. The admin reviews each offer and either accepts or rejects it via a PATCH /api/offers/:id request. If the admin accepts an offer, the customer is notified and directed to the payment module to complete the transaction at the agreed price. Once payment is confirmed, the order is created, the gem is marked as sold, removed from the inventory, and the listing is closed. All other pending offers on that same gem are automatically rejected. If the admin rejects an offer, the customer's offer status updates to Rejected and they can see this in their account.
When the admin needs to update a listing — such as changing the price or adding a new photo — they can edit it from the admin panel via PUT /api/marketplace/:id. If a listing needs to be removed without a sale, the admin can delete it via DELETE /api/marketplace/:id, which does not affect the inventory record.
4. Order Management & Tracking
The order tracking module manages everything that happens after a payment is confirmed and a purchase is finalised. It is built using Node.js, Express.js, and MongoDB, with the React Native frontend displaying a live status tracker to the customer. Every sale — whether through a direct purchase, a negotiated offer, or a won bid — automatically generates an order record the moment payment is confirmed through the payment module.
A unique order number is generated and assigned to each order. The order document in MongoDB stores the customer's details, which gem was purchased, the price paid, the payment reference ID, the purchase date, and the current status. The initial status is always set to Confirmed.
From the customer side, they can open their orders screen in the React Native app to see all current and past orders, fetched from GET /api/orders. Tapping on an order shows the full detail page fetched from GET /api/orders/:id, including the order number and a visual step-by-step progress tracker showing where the order currently sits in the delivery journey.
The order moves through four stages in sequence: Confirmed, Processing, Out for Delivery, and Delivered. The admin is responsible for manually advancing the order to the next stage as the physical delivery progresses. When the admin updates the status, this sends a PATCH /api/orders/:id request and the customer's order detail screen reflects the new stage the next time they open it.
If a customer wants to cancel their order before it has been dispatched, they can do so from their orders screen via DELETE /api/orders/:id. The admin can also cancel an order from the admin panel if needed. Once an order is marked as Delivered, it is considered complete and the customer becomes eligible to leave a review in the feedback module.
5. Feedback & Seller Rating System
The feedback module gives customers a way to leave a review after they have received a gem they purchased. It is built with Node.js, Express.js, and MongoDB on the backend and React Native on the frontend, and is linked directly to the order tracking module to ensure only genuine buyers can leave reviews.
After an order is marked as Delivered, the customer becomes eligible to leave a review for the gem they purchased. They navigate to the reviews section of the app, select the completed order, and are shown the review form. They choose a star rating from one to five and write a short comment. Submitting this sends a POST /api/reviews to the backend. The review is saved in MongoDB linked to both the specific gem and the specific order, which ensures that only customers who actually completed a purchase can leave a review — not anyone who simply browsed the listing.
Any customer visiting the marketplace can read the reviews on a gem's listing page. The reviews are fetched from GET /api/reviews/:gemId and displayed from most recent to oldest. Each review shows the reviewer's name, their star rating, and their comment.
The admin can view all reviews across the platform from the admin panel. If a review contains inappropriate content, the admin can delete it via DELETE /api/reviews/:id. Regular customers cannot delete or edit their own reviews once submitted. There are no reply threads and no editing — just straightforward honest feedback attached to each gem.
6. Bidding System
The bidding module handles gem listings that the admin puts up for auction-style bidding, where multiple customers compete by placing increasingly higher bids over a set period. This is separate from the negotiation system — rather than the admin reviewing private offers, bids are competitive and time-bound. It is built using Node.js, Express.js, and MongoDB on the backend and React Native on the frontend.
The bidding flow starts with the admin. When creating a bid listing, the admin selects a gem from the inventory, sets a starting bid price, and sets a bid end date and time. This is submitted to POST /api/bids and saved as a bid record in MongoDB linked to that gem. The listing becomes visible to all customers immediately and shows the current highest bid and the time remaining.
On the customer side, active bid listings are visible from a dedicated Bidding section in the React Native app, fetched from GET /api/bids. Each listing shows the gem photo, name, starting price, current highest bid, and the countdown to the end time. When a customer taps on a bid listing, they go to the detail page fetched from GET /api/bids/:id, which shows full gem details and the current bid status.
When a customer wants to place a bid, they enter an amount higher than the current highest bid and submit it. The app sends a POST /api/bids/:id/place request to the backend. The backend validates that the new bid is higher than the current highest bid and that the auction has not ended, then saves the new bid and updates the current highest bid on the listing. If a customer tries to place a bid lower than or equal to the current highest bid, the backend rejects it and returns an appropriate error message.
When the bid end time is reached, the listing is automatically closed and the customer with the highest bid is declared the winner. The winning customer is notified within the app and directed to the payment module to complete the transaction at their winning bid amount. Once payment is confirmed, an order is created in the order tracking module, the gem is removed from the inventory, and the bid listing is marked as closed. If no bids were placed before the end time, the admin is notified and can choose to relist the gem or return it to the inventory.
The admin can view all active and closed bid listings from the admin panel via GET /api/bids, and can delete a bid listing before it ends via DELETE /api/bids/:id if needed. Once a bid is closed and an order is created, the record is permanent and cannot be deleted.

7. Payment
The payment module handles all financial transactions that occur within the app. It is built using Node.js, Express.js, and MongoDB on the backend and React Native on the frontend, with Stripe integrated to process card transactions securely. No order is created in the system until a payment is successfully confirmed through this module.
Payment is triggered from three points in the app — a direct purchase from the marketplace, an accepted negotiation offer, or a won bid. In all three cases, the customer is redirected to the payment screen with the final amount pre-filled based on the context that triggered it. The customer sees the gem name, the amount due, and a form where they enter their card details including card number, expiry date, and CVV. The form is validated on the React Native frontend before submission to ensure no incomplete requests are sent to the backend.
When the customer submits, the app sends a POST /api/payments request to the backend. The backend forwards the payment details to Stripe for processing. Card details themselves are never stored in the MongoDB database — only the transaction result, the Stripe reference ID, the amount, the customer ID, the gem ID, and the payment status are saved as a payment document.
If the payment is successful, the backend returns a confirmation response, the payment document is saved with a status of Success, and the relevant module is notified to proceed — an order is created in the order tracking module, the gem is marked as sold, and the inventory is updated. If the payment fails — for example due to an incorrect card number or insufficient funds — the backend returns an error, no payment document is saved, no order is created, and the customer is shown a clear error message on the payment screen with the option to retry or cancel.
From the admin side, all completed payment records are visible in a payments overview screen in the admin panel, fetched from GET /api/payments. The admin can see the transaction reference, amount, customer name, gem purchased, and date for each record. An individual payment record can be fetched via GET /api/payments/:id. Payment records cannot be edited or deleted as they serve as a permanent financial record of all transactions on the platform.
